## Phase Requirements

| ID | Description | Research Support |

## Summary

Paraglide JS v2 (`@inlang/paraglide-js`) is a compiled, tree-shakable i18n library. Version 2 dropped the deprecated `@inlang/paraglide-sveltekit` adapter in favour of a framework-agnostic Vite plugin. Integration with SvelteKit requires four files: the Vite plugin config, `project.inlang/settings.json`, `src/app.html` with a `%lang%` placeholder, and a middleware wired into the existing `hooks.server.ts` via SvelteKit's `sequence` helper.

The cookie strategy (`PARAGLIDE_LOCALE` cookie, ~400-day max-age) satisfies I18N-03 with zero custom code. Adding `preferredLanguage` before `cookie` in the strategy array automatically reads `Accept-Language` on the server for first-time visitors (I18N-08). Switching locale with `setLocale()` triggers a full page reload by default, which re-runs the middleware and re-replaces `%lang%` — this is intentional in v2's design and is the correct pattern.

Sonner toast is added via the shadcn-svelte CLI command, which adds `svelte-sonner` as a dependency and drops a `Toaster` wrapper component into `$lib/components/ui/sonner/`. Placing `<Toaster />` once in `+layout.svelte` makes `toast()` callable from any component without prop drilling or a store.

**Primary recommendation:** Use `strategy: ['cookie', 'preferredLanguage', 'baseLocale']` in the Vite plugin (no URL prefix), wire `paraglideMiddleware` into the existing `hooks.server.ts` via `sequence`, add `%lang%` to `app.html`, and install Sonner via the shadcn-svelte CLI.

## Architecture Patterns

### Recommended Project Structure

```
project.inlang/
└── settings.json           # locales: ["en","de"], baseLocale: "en"
messages/
├── en.json                 # source messages (flat snake_case keys)
└── de.json                 # German translations
src/
├── app.html                # lang="%lang%" placeholder
├── hooks.server.ts         # sequence(existingHandle, paraglideHandle)
├── hooks.ts                # reroute: deLocalizeUrl (only if url strategy used)
└── lib/
    ├── paraglide/          # generated by Vite plugin (DO NOT EDIT)
    │   ├── messages.js
    │   ├── runtime.js
    │   └── server.js
    └── components/
        ├── ui/
        │   └── sonner/     # generated by shadcn-svelte add sonner
        └── layout/
            └── AppSidebar.svelte  # add LanguageSwitcher here
```

### Pattern 1: Vite Plugin Configuration (no URL prefix)

**What:** Register the Paraglide Vite plugin with a cookie-first + browser-detect strategy
**When to use:** Always — this is the entry point for the compiler

```typescript
// vite.config.ts
// Source: https://github.com/opral/paraglide-js/blob/main/examples/sveltekit/vite.config.ts
import { paraglideVitePlugin } from '@inlang/paraglide-js';

export default defineConfig({
  plugins: [
    tailwindcss(),
    sveltekit(),
    paraglideVitePlugin({
      project: './project.inlang',
      outdir: './src/lib/paraglide',
      strategy: ['cookie', 'preferredLanguage', 'baseLocale']
      // No 'url' strategy = no URL prefixes (/de/...) needed
    }),
    ...(process.env.VITEST ? [stubBunForTests()] : [])
  ],
  // ... rest of existing config
});
```

### Pattern 2: Server Middleware via sequence

**What:** Integrate paraglide middleware into the existing `hooks.server.ts` without replacing it
**When to use:** Required — the existing handle already does session parsing and CORS

```typescript
// src/hooks.server.ts
// Source: https://inlang.com/m/gerre34r/library-inlang-paraglideJs/sveltekit
import { sequence } from '@sveltejs/kit/hooks';
import { paraglideMiddleware } from '$lib/paraglide/server';

// Existing handle stays untouched
const existingHandle: Handle = async ({ event, resolve }) => {
  // ... session parsing, CSRF, CORS (unchanged)
};

const paraglideHandle: Handle = ({ event, resolve }) =>
  paraglideMiddleware(event.request, ({ request: localizedRequest, locale }) => {
    event.request = localizedRequest;
    return resolve(event, {
      transformPageChunk: ({ html }) => html.replace('%lang%', locale)
    });
  });

// paraglide MUST run after session so locals.user is already populated
export const handle: Handle = sequence(existingHandle, paraglideHandle);
```

### Pattern 3: HTML lang Placeholder

**What:** Replace hard-coded `lang="en"` in `app.html` with a server-replaced placeholder
**When to use:** Always — required for I18N-07

```html
<!-- src/app.html -->
<html lang="%lang%">
```

The middleware replaces `%lang%` with the detected locale string (e.g., `de`) on every SSR render.

### Pattern 4: reroute Hook (only if url strategy is used)

**What:** Strip locale prefixes from URLs so SvelteKit routing still works
**When to use:** ONLY if `url` is in the strategy array. Since we use cookie-first with no URL prefix, this file is NOT needed for this project.

Note: Because we use `strategy: ['cookie', 'preferredLanguage', 'baseLocale']` without `url`, there are no locale URL prefixes to strip, and `src/hooks.ts` with `deLocalizeUrl` is not required.

### Pattern 5: Language Switcher Component

**What:** A button/select inside AppSidebar that calls `setLocale()` to switch language
**When to use:** Required for I18N-01 and I18N-09

```svelte
<!-- src/lib/components/LanguageSwitcher.svelte -->
<script lang="ts">
  // Source: https://inlang.com/m/gerre34r/library-inlang-paraglideJs/runtime
  import { getLocale, setLocale, locales } from '$lib/paraglide/runtime.js';

  const nativeNames: Record<string, string> = {
    en: 'English',
    de: 'Deutsch'
  };

  const current = $derived(getLocale());

  function switchLocale(locale: string) {
    // setLocale triggers a full page reload by default (v2 deliberate design)
    setLocale(locale as 'en' | 'de');
  }
</script>

{#each locales as locale}
  <button
    onclick={() => switchLocale(locale)}
    aria-pressed={current === locale}
  >
    {nativeNames[locale] ?? locale}
  </button>
{/each}
```

### Pattern 6: Toaster in Root Layout

**What:** Place the Sonner `<Toaster />` once in the root layout so toasts are globally available
**When to use:** Required — must be in `+layout.svelte`

```svelte
<!-- src/routes/+layout.svelte -->
<script lang="ts">
  import { Toaster } from '$lib/components/ui/sonner/index.js';
  // ... existing imports
</script>

<Tooltip.Provider>
  <div class="h-screen bg-slate-100 flex flex-col overflow-hidden" style="height: 100dvh;">
    {@render children()}
  </div>
  <AppSidebar />
</Tooltip.Provider>

<Toaster />
```

Triggering a toast from any component:

```svelte
<script lang="ts">
  import { toast } from 'svelte-sonner';
</script>

<button onclick={() => toast.success('Saved!')}>Save</button>
```

### Anti-Patterns to Avoid

- **Using `@inlang/paraglide-sveltekit`:** Deprecated. The Vite plugin in `@inlang/paraglide-js` replaces it entirely.
- **Editing generated files in `src/lib/paraglide/`:** The Vite plugin regenerates them on every build. Put nothing hand-written there.
- **Calling `setLocale('de', { reload: false })`:** Without a reload, the SSR-injected `%lang%` and existing translated strings on the page will not update. Always use the default (reload: true) for a correct user experience.
- **Using `history.pushState` / `history.replaceState` for locale switching:** Conflicts with SvelteKit's router. `setLocale()` does its own reload.
- **Adding the `url` strategy without also adding `hooks.ts` with `deLocalizeUrl`:** Missing the reroute causes 404s for localized URLs. Since we use no URL strategy, this is avoided entirely.
- **Placing `<Toaster />` inside a conditional block:** Must be at the root layout level, always mounted, for position and portal to work correctly.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Cookie read/write for locale | Custom `document.cookie` manipulation | `cookie` strategy in Paraglide plugin | Paraglide handles `Set-Cookie` in SSR response headers, SameSite, domain scope, and max-age |
| Accept-Language parsing | Custom header parsing in `hooks.server.ts` | `preferredLanguage` strategy | Handles BCP-47 tags, quality values, fallback matching, and `navigator.languages` on client |
| Toast portal / z-index stacking | Custom overlay component | `svelte-sonner` via shadcn-svelte | Handles viewport stacking, keyboard dismiss, queue management, animation, and accessibility |
| HTML lang attribute injection | `svelte:head` with a reactive variable | `%lang%` + `transformPageChunk` in middleware | Runs server-side before HTML is sent; `svelte:head` can't update `<html>` attributes |
| Type-safe translation keys | TypeScript interface for message keys | Generated `m.*` functions from Paraglide compiler | Compiler produces fully typed message functions; adding new keys updates types automatically |

**Key insight:** The Paraglide v2 compiler runs at build/dev time and generates all runtime code. The "hard" i18n problems (SSR locale propagation, hydration, bundle splitting per locale) are solved by the compiler and middleware pattern — not by runtime state management.

## Common Pitfalls

### Pitfall 1: Running `paraglideHandle` Before the Existing Handle

**What goes wrong:** If `paraglideHandle` runs first in `sequence(paraglideHandle, existingHandle)`, `event.locals.user` is not yet populated when paraglide middleware executes. This does not break locale detection, but if future code in paraglide-related hooks needs the user, it will fail silently.
**Why it happens:** `sequence` runs handlers in order; session parsing is in the existing handle.
**How to avoid:** Always use `sequence(existingHandle, paraglideHandle)`.
**Warning signs:** `event.locals.user` is `null` inside `paraglideHandle` callback.

### Pitfall 2: Forgetting `%lang%` in `app.html`

**What goes wrong:** `transformPageChunk` replaces `%lang%` with the locale string, but if `app.html` still has `lang="en"`, the HTML `lang` attribute never changes. This breaks I18N-07.
**Why it happens:** `app.html` is not a Svelte file; it's a raw HTML template. The replacement is string-based, not reactive.
**How to avoid:** Change `<html lang="en">` to `<html lang="%lang%">` in `src/app.html`.
**Warning signs:** Browser DevTools shows `lang="en"` even after switching to German.

### Pitfall 3: `src/lib/paraglide/` Checked into Git

**What goes wrong:** The generated output directory gets checked into version control, causing merge conflicts and stale generated code.
**Why it happens:** `bunx @inlang/paraglide-js init` may not add it to `.gitignore`.
**How to avoid:** Add `src/lib/paraglide` to `.gitignore` after running init.
**Warning signs:** `git status` shows changes in `src/lib/paraglide/` after editing `messages/`.

### Pitfall 4: PWA Service Worker Caches Stale Locale

**What goes wrong:** The existing service worker caches API routes with network-first, but if the page HTML (which contains the baked-in `%lang%` value) is cached cache-first, a locale switch reloads to the cached old locale.
**Why it happens:** The service worker in `service-worker.ts` uses cache-first for HTML (app shell). After `setLocale()` triggers a reload, the service worker may serve the cached version.
**How to avoid:** Confirm the service worker uses network-first for navigation (HTML documents). Check `src/service-worker.ts` — if it precaches HTML, consider adding a cache-bust or ensuring the `transformPageChunk` locale doesn't end up in the precached shell.
**Warning signs:** After switching locale, the page appears correct, but after going offline and reloading, it reverts to the previous locale.

### Pitfall 5: Import Path Must Match `outdir`

**What goes wrong:** After init, Paraglide generates files into `outdir`. If `outdir` is set to `./src/lib/paraglide`, all imports must be from `$lib/paraglide/runtime.js` (with `.js` extension). Importing without the extension or from the wrong path causes module-not-found errors.
**Why it happens:** The generated files use `.js` extensions for ESM compliance.
**How to avoid:** Always import `from '$lib/paraglide/runtime.js'` and `from '$lib/paraglide/messages.js'` — with the `.js` extension.
**Warning signs:** TypeScript errors `Cannot find module '$lib/paraglide/runtime'`.

### Pitfall 6: `setLocale` in SSR Context Affects All Users

**What goes wrong:** In SvelteKit SSR, calling `setLocale()` in a `+page.server.ts` `load` function sets the locale on the shared global state, leaking one user's locale into another's response.
**Why it happens:** The `globalVariable` strategy uses a module-level variable, which is shared across concurrent requests in a single Node process.
**How to avoid:** Never call `setLocale()` in server-side load functions. The middleware (`paraglideMiddleware`) is the only correct place to set the server-side locale. The `serverAsyncLocalStorage` in the generated server file handles per-request isolation.
**Warning signs:** Users randomly see each other's locale in SSR responses under load.

## Open Questions

1. **PWA service worker interaction with locale-in-HTML**
   - What we know: The service worker does cache-first for the app shell HTML. `setLocale()` reloads the page, and the server re-renders with the new locale via `transformPageChunk`.
   - What's unclear: Whether the precache entry for the root HTML gets invalidated when locale changes, or whether a cached HTML shell with `lang="en"` gets served for offline users after they switched to German.
   - Recommendation: During the implementation task, inspect `src/service-worker.ts` and verify navigation requests go through network-first. If not, the `%lang%` baked into cached HTML will be wrong offline. This may be acceptable for a first phase since offline locale accuracy is a deferred concern.

2. **Exact `setLocale` type signature for TypeScript**
   - What we know: `setLocale` accepts a locale string and optional `{ reload: boolean }`. The locale must be one of the configured values.
   - What's unclear: Whether the generated type is `setLocale(locale: 'en' | 'de', options?: { reload?: boolean }): void | Promise<void>` or uses a different shape.
   - Recommendation: After running init, inspect the generated `runtime.js` and its companion `.d.ts` file to confirm the exact type before writing the language switcher.
